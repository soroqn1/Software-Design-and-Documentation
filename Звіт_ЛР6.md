# ДОКУМЕНТАЦІЯ ПРОГРАМНОГО ЗАБЕЗПЕЧЕННЯ
**(Прототип месенджера з підтримкою прикріплення файлів)**

## 1. ДОКУМЕНТАЦІЯ ВИМОГ

### 1.1. Призначення продукту
Розроблюваний програмний продукт є прототипом серверної частини (backend) месенджера. Основна мета застосунку — забезпечення швидкого та безпечного обміну текстовими повідомленнями та файлами (прикріпленнями) між користувачами через REST API.

### 1.2. Функціональні вимоги
Система повинна забезпечувати виконання наступних функцій:
* **Керування користувачами:** реєстрація нових користувачів з унікальними іменами; отримання інформації про користувача.
* **Керування діалогами:** створення особистих (direct) діалогів; перегляд історії повідомлень у конкретному діалозі.
* **Обмін текстовими повідомленнями:** відправка текстових повідомлень у створені діалоги; зміна статусу повідомлення (SENT, DELIVERED, READ).
* **Робота з файлами (вкладеннями):** 
  * Запит на створення слота для завантаження файлу (повертає унікальне посилання).
  * Завантаження файлу на сервер.
  * Автоматичне присвоєння файлу статусу "PENDING", а після перевірки — "SAFE" або "QUARANTINED".
  * Прикріплення безпечного файлу до повідомлення.
  * Отримання посилання на завантаження безпечного вкладення.

### 1.3. Нефункціональні вимоги
* **Продуктивність:** база даних повинна працювати у режимі WAL (Write-Ahead Logging) для забезпечення високої швидкості читання/запису.
* **Безпека:** файли не можуть бути завантажені іншими користувачами, якщо вони не пройшли перевірку (статус "SAFE").
* **Надійність:** обробка помилок та валідація вхідних даних (наприклад, заборона створення користувача з порожнім іменем).

## 2. ДОКУМЕНТАЦІЯ З АРХІТЕКТУРИ

### 2.1. Загальна архітектура системи
Застосунок побудований за класичною клієнт-серверною архітектурою. Серверна частина реалізована мовою TypeScript з використанням фреймворку Fastify. Взаємодія з клієнтами відбувається за допомогою REST API. 

Основні компоненти:
1. **Routing Layer (Маршрутизація):** обробка HTTP-запитів (`src/api/routes.ts`).
2. **Service Layer (Бізнес-логіка):** сервіси для роботи з користувачами, повідомленнями та файлами (`UserService`, `MessageService`, `AttachmentService`).
3. **Data Access Layer (Шар даних):** взаємодія з реляційною базою даних SQLite (`src/storage/database.ts`).
4. **File Storage (Файлове сховище):** локальна директорія `/uploads` для збереження фізичних файлів.

### 2.2. Архітектура бази даних
Реляційна модель бази даних складається з наступних основних сутностей:
* `users` — зберігає ідентифікатори та імена користувачів.
* `conversations` — зберігає дані про діалоги (тип: direct або group).
* `participants` — зв'язуюча таблиця між користувачами та діалогами (багато-до-багатьох).
* `messages` — зберігає текст повідомлення, статуси доставки та посилання на відправника, діалог і прикріплення (за наявності).
* `attachments` — зберігає метадані файлів (ім'я, MIME-тип, розмір, статус перевірки `scanStatus`).

## 3. ТЕХНІЧНА ДОКУМЕНТАЦІЯ (СПЕЦИФІКАЦІЯ)

### 3.1. Використані технології
* **Мова програмування:** TypeScript / Node.js.
* **Веб-фреймворк:** Fastify.
* **База даних:** SQLite (бібліотека `better-sqlite3`).
* **Тестування:** Vitest.

### 3.2. Специфікація API (основні ендпоінти)
1. `POST /users` — створення користувача. Тіло запиту: `{ "name": "string" }`.
2. `POST /conversations` — створення діалогу. Тіло: `{ "type": "direct", "participantIds": ["id1", "id2"] }`.
3. `POST /messages` — відправка повідомлення. Тіло: `{ "conversationId": "id", "senderId": "id", "text": "string", "attachmentId": "id (опціонально)" }`.
4. `POST /attachments/upload-url` — запит слота на завантаження. Тіло: `{ "filename": "string", "mimeType": "string", "sizeBytes": number }`.
5. `PUT /attachments/:id/upload` — фізичне завантаження бінарних даних файлу.
6. `GET /attachments/:id/url` — отримання посилання на завантаження перевіреного файлу.

### 3.3. Інструкція з розгортання
Для запуску системи на локальному середовищі необхідно:
1. Встановити залежності: `npm install`.
2. Запустити сервер у режимі розробки: `npm run dev`.
3. Виконати автоматизовані тести для перевірки працездатності: `npm run test`.
Сервер за замовчуванням запускається на порту `3000`.

## 4. ДОКУМЕНТАЦІЯ КОРИСТУВАЧА (ПОСІБНИК)

### 4.1. Вступ та початок роботи
Даний посібник призначений для розробників та тестувальників, що використовують REST API системи (наприклад, через Postman). 
Для початку роботи необхідно створити профіль користувача, відправивши `POST`-запит на `/users` з вашим іменем. У відповідь ви отримаєте унікальний `id`, який слід використовувати у подальших діях.

### 4.2. Створення діалогу та обмін повідомленнями
1. Знайдіть `id` співрозмовника.
2. Створіть діалог (`POST /conversations`), передавши ваш `id` та `id` співрозмовника.
3. Використовуйте отриманий `conversationId` для відправки повідомлень (`POST /messages`).

### 4.3. Відправка файлів
Щоб надіслати файл, виконайте три кроки:
1. Зареєструйте файл у системі (`POST /attachments/upload-url`), вказавши його назву та розмір. Ви отримаєте `attachmentId` та спеціальне посилання (`uploadUrl`).
2. Відправте сам файл за допомогою `PUT` запиту на отримане посилання `uploadUrl`. Система автоматично перевірить файл і призначить йому статус "SAFE".
3. Надішліть повідомлення (`POST /messages`), вказавши текст та `attachmentId`. 

*Увага: Неможливо надіслати повідомлення з файлом, який ще не був повністю завантажений або визнаний небезпечним (знаходиться у статусі PENDING або QUARANTINED).*

## 5. МАРКЕТИНГОВА ДОКУМЕНТАЦІЯ

### 5.1. Позиціонування продукту
**"SecureMessenger: Ваші дані під надійним захистом"**
Наш продукт — це сучасний, швидкий та мінімалістичний бекенд для корпоративних месенджерів. Він створений для компаній, які цінують швидкість комунікації, але не готові йти на компроміси щодо безпеки передачі даних.

### 5.2. Цільова аудиторія
* Корпоративні клієнти та малий/середній бізнес.
* Команди розробників, яким потрібна надійна REST API платформа для створення власних клієнтських додатків (Mobile, Web, Desktop).
* Організації з високими вимогами до безпеки вкладень.

### 5.3. Унікальна торгова пропозиція (УТП)
1. **Премодерація файлів:** Вбудована система безпеки на рівні API. Файл ніколи не потрапить до отримувача, доки не отримає статус "SAFE". Небезпечні файли автоматично відправляються в карантин.
2. **Висока швидкодія:** Використання SQLite з налаштуванням WAL-журналювання гарантує миттєвий відгук бази даних навіть під час активного обміну повідомленнями.
3. **Легка інтеграція:** Чистий REST API підхід з чітко розділеними процесами (завантаження файлів відділене від відправки повідомлень) дозволяє легко інтегрувати месенджер у будь-яку існуючу інфраструктуру.
